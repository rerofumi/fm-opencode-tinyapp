# 開発エージェントのための作業方針

このドキュメントは、開発エージェントがプロジェクトで作業を進める上での基本的な方針を定めたものです。

## Git ワークフロー規約

### 1. ブランチ戦略
- `main`: 本番ブランチ
- `develop`: 開発ブランチ
- `feature/*`: 機能開発ブランチ
- `bugfix/*`: バグ修正ブランチ
- `release/*`: リリース準備ブランチ

### 2. コミットメッセージ
コミットメッセージは Conventional Commits 規約に従います。
例:
- `feat`: 新機能の追加
- `fix`: バグ修正
- `docs`: ドキュメントの変更
- `style`: コードフォーマットの変更
- `refactor`: リファクタリング
- `test`: テストの追加・修正
- `chore`: ビルドプロセスや補助ツールの変更


## ドキュメント規約

### 1. コードドキュメント
- 公開されるインターフェース、型、関数には、その目的と使用方法を説明するコメントを記述します。
- 複雑なロジックやアルゴリズムには、その動作を説明するコメントを追加します。

### 2. API ドキュメント
- APIの仕様は OpenAPI などの標準的なフォーマットで記述します。
- 各エンドポイントには、目的、パラメータ、リクエスト/レスポンスの例を記載します。

### 3. README
- プロジェクトの概要
- セットアップとインストールの手順
- 基本的な使用方法
- 貢献の方法に関するガイドライン

## 品質保証規約

### 1. コードレビュー
コードレビューでは、以下の点を確認します。
- 機能要件を満たしているか
- エラーハンドリングが適切か
- パフォーマンス上の問題がないか
- セキュリティ上の脆弱性がないか
- テストが十分に書かれているか
- ドキュメントが更新されているか

### 2. リリースプロセス
1. 機能開発とテストの完了
2. コードレビューと承認
3. `develop` ブランチへのマージ
4. リリースブランチの作成
5. 本番環境へのデプロイ
6. バージョンタグの付与とリリースノートの作成

### 3. 監視
- アプリケーションのエラーログを監視します。
- パフォーマンスに関するメトリクスを収集・分析します。
- ユーザーからのフィードバックを収集し、改善に役立てます。

### 4. 動作確認
- バックエンドやサーバーサービス(npm run dev 含む)は起動したら終了しないため、動作確認を目的として起動しないこと
- バックエンドやサーバーサービスはビルド時のエラー確認にとどめること
- ユーザーからの指示があった場合のみバックエンドやサーバーサービスの起動を行うが、その際はバックグラウンドプロセスとして起動し、コンソールはリダイレクトによるロギングを必ず行う事


## 開発環境規約

### 1. mise の利用
- mise を使用し、全ての環境、起動は mise を通じて行う
- mise.toml を作成し、開発に必要な tools や動作確認のための task 定義を行う
- mise exec による実行、起動は避け、task 定義をしてからその task 実行を行うようにする
